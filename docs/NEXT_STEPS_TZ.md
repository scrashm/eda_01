# ТЗ на дальнейшую разработку

## 1. Контекст и цель

Продукт уже реализует базовый пользовательский поток: онбординг, случайный выбор страны, просмотр рецептов, избранное, история, фильтрацию и подбор по ингредиентам.  
Цель следующего этапа (`v0.2`) — довести приложение до стабильного ежедневного использования: убрать критичные UX-дефекты, усилить качество данных, повысить надежность и покрытие тестами.

## 2. Scope релиза `v0.2`

### Входит в релиз

- исправление критичных визуальных багов и рисков запуска;
- улучшение рецептурных сценариев (готовка, покупки, заметки);
- техническая подготовка Android-пайплайна к новым требованиям Flutter;
- расширение тестирования и базовой аналитики.

### Не входит в релиз

- монетизация и рекламные механики;
- backend и авторизация;
- публикация в сторах как часть этого ТЗ.

## 3. Функциональные требования

### FR-1. Экран случайной страны (стабильность UI)

**Проблема:** при запуске на Windows зафиксирован `RenderFlex overflow` в `random_country_screen`.  
**Требование:** на поддерживаемых форм-факторах не должно быть overflow-ошибок и обрезанного контента.

**Критерии приемки:**

- отсутствуют `RenderFlex overflow` при запуске на desktop и на типичных мобильных разрешениях;
- карточки shortcut корректно адаптируются по высоте/типографике;
- в debug-логе нет повторяющихся layout exceptions для главного экрана.

### FR-2. Улучшение экрана рецепта

**Требование:** завершить сценарий приготовления в одном экране.

**Доработки:**

- кнопка "Добавить ингредиенты в список покупок" на экране рецепта;
- блок "Мои заметки" (текстовая заметка, рейтинг 1-5, флаг "готовил");
- визуальная индикация сохраненного состояния.

**Критерии приемки:**

- пользователь может изменить заметку/рейтинг/статус и увидеть сохранение после перезапуска;
- добавление ингредиентов не создает дубликаты по ключу ингредиента;
- список покупок отражает количество совпадений и варианты записей.

### FR-3. Контент и релевантность выдачи

**Требование:** расширить dataset для более стабильного подбора.

**Доработки:**

- увеличить базу до 30+ стран;
- для каждой страны минимум 3 блюда;
- для каждого блюда 3 уровня сложности с заполненными метаданными (`type`, `tags`, `spiceLevel`, `exoticLevel`, `pantryKeywords`).

**Критерии приемки:**

- фильтры не дают "пустую выдачу" в типичных комбинациях чаще 10% случаев;
- для каждой страны доступны карточки блюд и рецептов без пустых полей;
- `flutter analyze` без ошибок после обновления данных.

### FR-4. Поиск и фильтры

**Требование:** улучшить предсказуемость фильтрации и поиска.

**Доработки:**

- нормализованный поиск по названию/стране/ингредиентам (case-insensitive, trim);
- единые словари тегов и типов;
- отображение активных фильтров в UI с возможностью быстрых сбросов.

**Критерии приемки:**

- поиск выдает ожидаемые результаты по частичному совпадению;
- сброс фильтров восстанавливает полный список;
- пользователь видит количество найденных результатов при каждом изменении фильтра.

## 4. Технические требования

### TR-1. Android build readiness

- миграция app-модуля на Built-in Kotlin по гайду Flutter;
- проверка зависимостей, которые тянут устаревший KGP;
- обновление `shared_preferences`/`shared_preferences_android` до совместимых версий при доступности.

**Критерии приемки:**

- предупреждение Flutter о migration risk для app-модуля устранено;
- проект собирается командой `flutter build apk` без новых критичных предупреждений.

### TR-2. Архитектурная эволюция состояния

- декомпозиция `AppState` на доменные сервисы (навигация не входит);
- выделение интерфейса персистентности поверх `SharedPreferences`;
- подготовка к миграции на `Riverpod` или `Provider` без big-bang рефактора.

**Критерии приемки:**

- минимум 2 независимых state-модуля вынесены из монолита;
- бизнес-методы тестируются без запуска UI.

### TR-3. Наблюдаемость и аналитика

- добавить легковесный слой событий (без внешней отправки на этом этапе);
- фиксировать ключевые действия: запуск, реролл страны, открытие рецепта, избранное, фильтры, добавление в список покупок.

**Критерии приемки:**

- события централизованно описаны и вызываются из ключевых точек;
- при отключенном отправителе приложение работает без деградации.

## 5. Нефункциональные требования

- время первого открытия главного экрана <= 2.5 сек на среднем Android-устройстве;
- отсутствие падений на базовом smoke-сценарии (10 последовательных запусков);
- поддержка offline-режима с локальным JSON как source of truth.

## 6. Тестирование

### Обязательное

- unit-тесты: парсинг данных, фильтры, агрегатор списка покупок, журнал рецептов;
- widget-тесты: онбординг, главный экран, экран рецепта, список покупок;
- smoke integration: запуск -> выбор страны -> открытие рецепта -> добавление в избранное.

### Желательное

- golden tests для главного экрана и карточки рецепта;
- regression-тест на отсутствие overflow в compact layout.
